项目目前是一个“林业知识库 + 检索增强生成（RAG）问答系统 + 论文实验管理工程”的早期版本。

更通俗地说：

- 你有一些林业教材或资料。
- 你把教材内容切成一条条“知识块”。
- 系统根据用户问题，从知识块里找证据。
- 再让大语言模型只根据这些证据回答。
- 最终论文要证明：这种方法在林业知识问答中是否更可追溯、更少胡编、能在证据不足时拒答。

当前第一阶段只严肃处理两本已经切块的书：

- `林木病理学（第3版）`：1291 条知识块。
- `园林植物遗传育种（第3版）`：457 条知识块。

总计 1748 条知识块。这个数字来自 `experiments/validation_latest.json` 的实际校验结果。

## 2. 最重要的科研原则

这个项目以后要走 SCI 投稿路线，所以必须先守住底线：

- 不编造文献。
- 不编造页码。
- 不编造数据。
- 不编造实验结果。
- 不把未审核的自动结果当作论文结论。
- 不公开可能受版权限制的原始教材全文或大段原文。
- 任何论文表格中的数字，都必须能追溯到 `experiments/` 中的实验结果文件。

你要把这个项目当成“证据链工程”，而不是只做一个能聊天的 demo。

## 11. `forestryllm/`：新的可复现实验代码

这是后续论文实验的主代码目录。

### `forestryllm/__init__.py`

作用：告诉 Python 这是一个包。

里面记录了当前版本号。

### `forestryllm/__main__.py`

作用：让你可以这样运行项目：

```bash

```

### `forestryllm/cli.py`

作用：命令行入口。

它定义了这些命令：

- `validate-corpus`：校验知识库字段和数据质量。
- `convert-excel`：把切块 Excel 转成 CSV/JSONL。
- `build-index`：用 bge-m3 建 Qdrant 向量库。
- `retrieve`：对一个问题检索证据。
- `answer`：检索证据后回答问题。
- `evaluate`：对评测集跑指标。
- `export-paper-tables`：把实验结果导出为论文表格 CSV。

### `forestryllm/config.py`

作用：读取配置文件，并把相对路径转换成绝对路径。

通俗理解：它帮助程序知道数据、模型和实验文件在哪里。

### `forestryllm/corpus.py`

作用：处理语料。

它负责：

- 读取 JSONL 知识块。
- 合并多个语料文件。
- 校验字段是否完整。
- 检查页码是否合法。
- 检查 `chunk_id` 是否重复。
- 把 Excel 转成 CSV/JSONL。

### `forestryllm/retrieval.py`

作用：提供一个简单的词面检索方法。

它不是正式的高质量检索器，而是 smoke 测试用的轻量方法。优点是不需要加载大模型，方便快速测试流程。

### `forestryllm/index.py`

作用：处理正式向量索引。

它负责：

- 加载 bge-m3 embedding 模型。
- 生成知识块向量。
- 写入 Qdrant。
- 使用 Qdrant 做向量检索。

这个文件涉及较重的运行环境。没有准备好模型、GPU/CPU 环境或 Qdrant 数据时，不要随便运行 `build-index`。

### `forestryllm/rag.py`

作用：RAG 问答逻辑。

它负责：

- 构造提示词。
- 要求模型只根据证据回答。
- 证据不足时输出固定拒答句。
- 调用 Ollama。
- 从检索结果里生成引用列表。

### `forestryllm/evaluation.py`

作用：评测检索和拒答。

它会计算：

- `recall_at_k`
- `mrr`
- `ndcg`
- `rejection_accuracy`
- `citation_hit_rate`

注意：它不会自动计算 `answer_support_rate` 和 `unsupported_generation_rate`，因为“答案是否真正被证据支持”需要人工审核或可靠评审流程，不能随便让程序猜。

### `forestryllm/paper.py`

作用：把实验结果导出成论文可用的 CSV 表格。

目前主要把 `experiments/latest_evaluation.json` 里的指标导出到 `paper/tables/latest_metrics.csv`。

### `forestryllm/__pycache__/`

作用：Python 自动生成的缓存文件夹。

里面的 `.pyc` 文件是 Python 为了加快运行生成的，不是你手写的研究材料。通常不用管，也不应该作为论文内容。

### 第一步：先学会跑通项目

### 第二步：人工核查当前唯一 warning

打开：

`research/audit_queue.csv`

找到：

`M_LMBLX_V3_0055`

然后回到：

- `Forestry_KB/exports_datas/B001_M_林木病理学/B001_M_林木病理学_chunks_clean.csv`
- 或原始书籍 PDF 对应页码

核查它为什么只有 14 个字。

你只能做三种处理：

1. 如果原书那里本来就是短句，保留，并在审核备注中说明。
2. 如果切块漏了内容，回到 Excel 工作表修正，再重新导出。
3. 如果这条不应作为独立知识块，记录原因，再决定是否合并或删除。

不要直接凭感觉改 JSONL。

### 第三步：建设正式评测集

这是 SCI 论文最重要的一步之一。

你需要从两个方向设计问题：

1. 可回答问题：答案能从知识块中找到明确证据。
2. 不可回答问题：当前知识库没有足够证据，系统应该拒答。

建议第一版正式评测集规模：

- 先做 50 条可回答问题。
- 再做 20 条不可回答问题。
- 每条问题都要填 `gold_chunk_ids`。
- 每条问题都要标注 `review_status`。

不要一开始就做几百条。先保证质量。

### 第四步：做专家或导师审核

你自己做的问题不能直接当强证据。

理想流程：

1. 你先生成候选问题和 gold evidence。
2. 你自己初审。
3. 请导师、同门或领域专家复核一部分或全部样本。
4. 记录审核人、日期、意见。

如果只有你一个人审核，论文中必须诚实说明这个限制。

### 第五步：跑正式检索实验

先不要急着让大模型生成答案。先评估检索。

需要比较至少两个检索方法：

- 简单词面检索：当前 `lexical_cosine_smoke` 可作为弱基线。
- bge-m3 + Qdrant 向量检索：应作为主要方法或强基线。

指标包括：

- Recall@k
- MRR
- nDCG
- Citation Hit Rate

注意：正式实验前必须确认 `models/bge-m3/` 中模型文件完整可用。

### 第六步：再做问答与拒答实验

检索稳定后，再让模型回答。

重点指标：

- 引用是否命中正确证据。
- 答案是否真的被证据支持。
- 证据不足时是否拒答。
- 是否出现无依据生成。

其中“答案是否被证据支持”必须人工审核，不能只靠程序自动判断。

### 第七步：开始写论文，但只写已验证内容

论文写作顺序建议：

1. 方法：先写方法，因为它来自你的实际流程。
2. 数据集和语料：写语料来源、字段、统计、合规边界。
3. 评测方案：写评测集构建和审核流程。
4. 结果：等正式实验跑完再写。
5. 讨论：诚实写限制。
6. 引言和摘要：最后写。

不要先写摘要，因为你现在还没有正式结果。

## 21. 最容易犯的错误

### 错误 1：把 demo 当论文结果

能问答不等于能发论文。论文需要评测集、指标、对照和可复现记录。

### 错误 2：把 smoke 测试当正式实验

`smoke_qa.jsonl` 只是用来检查流程能不能跑。它太小、未审核，不能作为论文主结果。

### 错误 3：盲目扩充语料

语料越多，不一定越好。低质量切块会降低检索和回答质量，还会增加审核成本。

### 错误 4：让模型自己证明自己正确

如果模型生成答案，再让模型判断答案是否正确，这种证据很弱。关键样本必须人工审核。

### 错误 5：引用不核验

SCI 论文中每条文献、页码、实验数字都必须可核验。不能凭印象写。

## 22. 当前最稳妥的论文题目方向

暂定方向可以是：

`面向可信林业问答的证据约束检索增强生成方法`

中文理解：

“面向可信林业问答的证据约束检索增强生成方法”

这个题目方向的核心不是发明一个全新大模型，而是证明：

- 林业专业知识可以被结构化成可追溯知识块。
- RAG 可以减少无依据回答。
- 引用锚点可以提高可核查性。
- 拒答机制可以降低证据不足时的错误回答风险。

等正式实验完成后，再决定题目是否需要收窄到“林木病理”或扩展到“林业教材知识库”。
